Developer Documentation

A RAG pipeline you can rewire, not rewrite

The RAG Module is a configurable Retrieval-Augmented Generation pipeline. Every stage — parsing, chunking, embedding, vector storage, and retrieval — is an independent, swappable plugin resolved from plain JSON config. Change one stage without touching the others.

Python ≥ 3.13 FastAPI HTTP surface Qdrant vector store Azure OpenAI embeddings Pydantic typed contracts uv packaging

1 Overview & Architecture #

Who this is for: developers evaluating the pattern, and engineers extending or operating the pipeline.

Most RAG systems begin life as a single script: read a file, split it, embed it, push it to a vector DB, query it. That works right up until the first real requirement — "we need PDF as well as DOCX", "we want section-aware chunking", "we need a different embedder in staging", "we might move off Qdrant". At that point a linear script becomes brittle.

This project answers that by treating each pipeline stage as a strategy. Strategies are resolved from configuration through factories, then orchestrated by a single RAGPipeline class. The payoff is three properties that a script cannot give you:

🧩 Isolation

Each stage has one clear job. Parsers own document understanding; chunkers own chunk formation and metadata; embedders own vectors; vector DBs own persistence; retrieval owns query-time behavior.

🔁 Replaceability

Swap an implementation by changing a config key — no edits to pipeline code. New backends are one register_*() call away.

🧪 Testability

Every strategy is unit-testable in isolation, or exercised end-to-end through the pipeline and the FastAPI surface.

Data flows #

There are two primary flows: indexing (ingest a document) and retrieval (answer a query). The pipeline only moves data between stages; it never parses, chunks, embeds, or searches itself.

Design patterns & key decisions #

Pattern / decisionWhereWhy it matters
Strategy + Registry + Factory factory/*.py Config keys ("docx", "qdrant"…) map to classes. The pipeline initializes from JSON instead of importing concrete classes.
Typed stage contracts Pydantic data_models.py DocumentParserResult, Chunk, ScoredChunk are the explicit hand-offs between stages, so a stage can be replaced as long as it honors the contract.
Structured parser output ParsedContent Parsers return mode: "text" | "sections", not a flat string. Rich parsers can expose sections; chunkers branch on parser intent. See ADR 0001.
Chunk vs ScoredChunk split retrieval/data_models.py A Chunk has query-independent identity; a ScoredChunk pairs it with a query-specific score. Only the vector DB and retrieval layers traffic in scores.
Selective error taxonomy errors.py A typed RAGError hierarchy names failures per stage (ParserRejected, EmbedderFailed…). See ADR 0002.
Atomic indexing abort rag_pipeline.indexer If any stage fails, indexing raises and writes nothing — no partial documents land in the vector DB. See ADR 0003.

The orchestrator: RAGPipeline #

memory_module/rag_pipeline.py is the heart of the system, and deliberately thin. Its constructor resolves each configured strategy through _resolve_component(); a stage whose <stage>_key is absent resolves to None and is simply unavailable. Two public methods drive the flows: indexer() and retrieve().

Design rule RAGPipeline must only orchestrate. If it starts accumulating parsing or chunking logic, the abstraction is drifting — push that logic back into the owning strategy.

How a component is resolved

python# config drives resolution; missing key => component is None
strategy_key    = config.get(f"{base_key}_key")      # e.g. "parser_key": "docx"
strategy_kwargs = config.get(f"{base_key}_kwargs", {})  # optional constructor kwargs
component       = factory(strategy_key, **strategy_kwargs)
# a bad key or a construction failure is normalized into ConfigError

Strategy plugins #

Parser docx

Turns an uploaded file into a DocumentParserResult. accepts() is a pre-flight check; convert() does the work.

Chunker document

Consumes parser output, produces normalized Chunks. Owns chunk IDs and chunk metadata.

Embedder azure_openai

embed(text) → dense vector. Provider details are hidden behind the interface.

Vector DB qdrant

Persistence + search: add_chunks, retrieve, delete.

Retrieval similarity

Query-time behavior on top of the vector DB. Receives a vector_db instance.

Parser — DocxParser

Implements DocumentParserBase. accepts() validates the extension via FileOps and records last_error on rejection. convert() walks the DOCX paragraphs, groups them into ParsedSections by heading style, computes a content-hash document_id (doc_<hash>), and emits mode="sections" when headings exist, otherwise mode="text".

Chunker — DocumentChunker

Wraps chonkie's TokenChunker (defaults: chunk_size=2048, chunk_overlap=128, tokenizer="character"). It chunks the full text in text mode, or per-section in sections mode. Chunk identity is a deterministic uuid5 of <document_id>_chunk_<i>, so re-indexing the same document overwrites rather than duplicates. Caller tags from request metadata flow onto each chunk.

Embedder — AzureEmbeddingGenerator

Uses the OpenAI SDK pointed at an Azure endpoint. embed(text) returns a list of vectors (one per input). The pipeline flattens a single-item batch — if embedding[0] is a list: embedding = embedding[0] — so both list-of-vector and single-vector embedders interoperate.

Vector DB — QdrantVectorMemory

Implements BaseVectorMemory. Stores each chunk as a Qdrant point keyed by chunk_id, with text, serialized metadata, and token_count in the payload. retrieve() translates {"metadata.tags": "finance"}-style filters into Qdrant FieldConditions and returns ScoredChunks carrying the raw similarity score.

Retrieval — SimilarityRetrievalStrategy

The thinnest strategy: it forwards the RetrievalRequest (embedding, top_k, filters) straight to vector_db.retrieve(). It exists as a seam so hybrid search, reranking, or fusion can be dropped in later without touching the pipeline or API.

Error taxonomy #

All pipeline failures subclass RAGError and carry a stable code string, so callers can branch on err.code rather than parsing messages. The FastAPI layer surfaces these as 400 responses with the message in detail.

ExceptioncodeRaised when
ConfigErrorconfig_errorConfig isn't a dict, a key isn't a string, an invalid key, or a strategy fails to construct.
ParserRejectedparser_rejectedparser.accepts(document) returns False (unsupported document type).
NoChunksProducedno_chunks_producedThe chunker returns an empty list for the document.
EmbedderFailedembedder_failedThe embedder raises while embedding a chunk or the query.
VectorDBFailedvector_db_failedThe vector DB raises during add_chunks or retrieval.
InvalidQueryinvalid_queryRetrieval is called with an empty / non-string query.
Parser rejection ≠ parser failure A rejection is a parser's pre-flight refusal to handle a document (accepts() → False). A failure is a parser that accepted a document and then could not parse it. The taxonomy keeps these distinct on purpose.

4 API Reference #

HTTP surface exposed by main.py. Base URL: http://localhost:8000.

Authentication The application itself ships without an auth layer — secure it behind your own gateway for anything beyond local use. Upstream credentials (Azure key, optional Qdrant key) are read from environment variables, never from the request.

Data models #

These Pydantic models are the contracts between stages and the shapes returned over HTTP.

python# ---- parser output ----
class ParsedSection(BaseModel):
    title: str | None = None
    text: str
    level: int | None = None
    metadata: dict[str, Any] | None = None

class ParsedContent(BaseModel):
    mode: Literal["text", "sections"]
    text: str
    sections: list[ParsedSection]

class FileMetadata(BaseModel):
    document_id: str
    document_title: str

class DocumentParserResult(BaseModel):
    content: ParsedContent
    file_metadata: FileMetadata | None = None

# ---- indexed content ----
class ChunkMetadata(BaseModel):
    document_id: str
    document_title: str | None = None
    tags: list[str] | None = None
    chunk_version: str | None = None
    created_at: datetime            # defaults to utcnow()

class Chunk(BaseModel):
    chunk_id: str
    text: str
    embedding: list[float] = []
    metadata: ChunkMetadata
    token_count: int | None = None

# ---- retrieval ----
class RetrievalRequest(BaseModel):
    query_text: str
    query_embedding: list[float]
    top_k: int = 5
    filters: dict | None = None

class ScoredChunk(BaseModel):
    chunk: Chunk
    score: float                    # raw, strategy-specific similarity score

Drive the pipeline directly in Python

pythonfrom memory_module.rag_pipeline import RAGPipeline

pipeline = RAGPipeline({
    "embedder_key": "azure_openai",
    "vector_db_key": "qdrant",
    "retrieval_key": "similarity",
})

results = pipeline.retrieve(
    query="what were the Q3 finance numbers?",
    top_k=3,
    filters={"metadata.tags": "finance"},
)

for scored in results:            # List[ScoredChunk]
    print(f"{scored.score:.4f}  {scored.chunk.text[:80]}")

Adding a new strategy #

The extension contract is intentionally small: implement one interface, register one key. Example — a PDF parser.

python# 1. implement the interface
from fastapi import UploadFile
from memory_module.parser.document_parser_base import DocumentParserBase
from memory_module.parser.data_models import DocumentParserResult

class PdfParser(DocumentParserBase):
    def accepts(self, file_stream: UploadFile) -> bool:
        return (file_stream.filename or "").lower().endswith(".pdf")

    def convert(self, file_stream: UploadFile) -> DocumentParserResult:
        ...  # return a DocumentParserResult

# 2. register one key
from memory_module.factory.parser_factory import register_parser
register_parser("pdf", PdfParser)

# 3. activate it via config — no pipeline or API changes needed
#    { "parser_key": "pdf", ... }

The same three steps apply to every stage via its factory: register_chunker, register_embedder, register_vector_db, register_retrieval_strategy. Honor the base interface and the typed contracts, and the rest of the pipeline needs no changes.

Ownership boundaries when extending Parsers own document understanding · chunkers own chunk formation and metadata · embedders own vectors · vector DBs own persistence · retrieval owns query-time behavior · RAGPipeline only orchestrates.
